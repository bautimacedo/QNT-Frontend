# v0.5.0 — Gestión de usuarios y aprobación (ADMIN)

**VERSIÓN:** v0.5.0  
**SLUG:** gestion-usuarios-admin  
**DEPENDENCIAS:** v0.1.1 (registro de usuario — ya existe flujo de registro y estados)  
**ESTIMACIÓN:** 3h–4h  
**PRIORIDAD:** alta (los usuarios registrados no pueden acceder sin aprobación admin)

---

## Origen

> Pedido del responsable del proyecto. Con el registro implementado (v0.1.1), los usuarios
> quedan en estado `PENDIENTE_APROBACION`. Se necesita un panel para que el ADMIN pueda
> aprobar, rechazar y gestionar usuarios. El backend ya tiene todos los endpoints:
> listar, listar pendientes, aprobar, enable/disable, asignar/quitar roles.

---

## Descripción

Implementar el **panel de gestión de usuarios para ADMIN**, con dos vistas principales:

1. **Usuarios pendientes** (`/usuarios-pendientes`): listado de usuarios con estado
   `PENDIENTE_APROBACION`, con acciones "Aprobar" (asigna rol y activa) y "Rechazar"
   (desactiva el usuario).

2. **Usuarios (todos)** (`/usuarios`): listado general de todos los usuarios del sistema,
   con filtro/búsqueda, y acciones: ver detalle, activar/desactivar, asignar/quitar roles.

Ambas vistas son **solo para ROLE_ADMIN**. Si un usuario sin ese rol intenta acceder,
redirigir o mostrar "Sin permisos".

### Contexto backend (INFORME_BACKEND_PARA_FRONTEND.md)

**Endpoints relevantes (§5.2 Usuarios, §5.3 Roles):**

| Método | Ruta | Descripción |
|--------|------|-------------|
| GET | `/usuarios` | Listar todos los usuarios (ADMIN) |
| GET | `/usuarios/pendientes` | Listar solo los pendientes de aprobación (ADMIN) |
| GET | `/usuarios/search?email=` | Buscar por email (ADMIN) |
| PUT | `/usuarios/{id}/aprobar` | Aprobar usuario pendiente (body: `AprobarUsuarioRequest`) (ADMIN) |
| PUT | `/usuarios/disable?email=` | Desactivar usuario (ADMIN) |
| PUT | `/usuarios/enable?email=` | Activar usuario (ADMIN) |
| PUT | `/usuarios/assign-role` | Asignar rol (body: `AssignRoleRequest`) (ADMIN) |
| PUT | `/usuarios/remove-role` | Quitar rol (body: `AssignRoleRequest`) (ADMIN) |
| GET | `/roles` | Listar roles (para select de aprobación/asignación) (ADMIN) |

**Modelos relevantes:**

- **Usuario (§6.1):** id, nombre, apellido, email, roles[], estado (`PENDIENTE_APROBACION` | `ACTIVO` | `DESACTIVADO`), activo, dni, etc.
- **Role (§6.2):** id, codigo, nombre.
- **AprobarUsuarioRequest (§6.5):** `{ roleCodigo: string }` — Valores típicos: `ROLE_ADMIN`, `ROLE_PILOTO`, `ROLE_USER`.
- **AssignRoleRequest (§6.11):** `{ email: string, roleCodigo: string }`.
- **EstadoUsuario (§10):** `PENDIENTE_APROBACION`, `ACTIVO`, `DESACTIVADO`.

### VISTA 1: Usuarios pendientes (`/usuarios-pendientes`)

**Propósito:** Cola de aprobación. El admin ve quién se registró y decide si aprobarlo
(asignándole un rol) o rechazarlo.

**Datos:** `GET /usuarios/pendientes` → array de usuarios con `estado: "PENDIENTE_APROBACION"`.

**Columnas de la tabla:**

| Columna | Campo | Notas |
|---------|-------|-------|
| Nombre | `nombre` + `apellido` | Concatenar |
| Email | `email` | |
| Fecha registro | — | Si el backend no devuelve fecha de creación, omitir esta columna |
| Acciones | — | Botones "Aprobar" y "Rechazar" |

**Acción "Aprobar":**
1. Click en "Aprobar" → abrir modal/dropdown con selector de rol.
2. Cargar roles desde `GET /roles` (select con opciones: ROLE_ADMIN, ROLE_PILOTO, ROLE_USER, etc.).
3. Admin selecciona un rol → click "Confirmar".
4. `PUT /usuarios/{id}/aprobar` con body `{ roleCodigo: "ROLE_PILOTO" }` (o el elegido).
5. Éxito → quitar usuario de la lista (o refrescar) + mensaje "Usuario aprobado".
6. Error → mostrar mensaje.

**Acción "Rechazar":**
1. Click en "Rechazar" → confirmación "¿Rechazar a [nombre]? No podrá acceder al sistema."
2. `PUT /usuarios/disable?email=[email]` → desactiva al usuario (estado → DESACTIVADO).
3. Éxito → quitar de la lista + mensaje "Usuario rechazado".
4. Error → mostrar mensaje.

**Estado vacío:** "No hay usuarios pendientes de aprobación."

### VISTA 2: Usuarios (todos) (`/usuarios`)

**Propósito:** Panel de gestión completa de usuarios del sistema.

**Datos:** `GET /usuarios` → array de todos los usuarios.

**Columnas de la tabla:**

| Columna | Campo | Notas |
|---------|-------|-------|
| Nombre | `nombre` + `apellido` | |
| Email | `email` | |
| Roles | `roles[].codigo` | Mostrar como badges/chips (ej. "ADMIN", "PILOTO") |
| Estado | `estado` | Badge con color: ACTIVO → verde, PENDIENTE_APROBACION → amarillo, DESACTIVADO → rojo |
| Activo | `activo` | Indicador visual (check o cross) |
| Acciones | — | Menú contextual o botones |

**Filtro/búsqueda:**
- Buscador por email (usa `GET /usuarios/search?email=` o filtro cliente sobre la lista).
- Filtro por estado: Todos / Activos / Pendientes / Desactivados (filtro cliente).

**Acciones por usuario (en la fila o menú contextual):**

| Acción | Cuándo se muestra | Endpoint |
|--------|-------------------|----------|
| Aprobar | Solo si `estado == PENDIENTE_APROBACION` | `PUT /usuarios/{id}/aprobar` (con modal de rol) |
| Activar | Solo si `estado == DESACTIVADO` | `PUT /usuarios/enable?email=` |
| Desactivar | Solo si `estado == ACTIVO` | `PUT /usuarios/disable?email=` |
| Asignar rol | Siempre (o si ACTIVO) | `PUT /usuarios/assign-role` (modal con select de rol) |
| Quitar rol | Si tiene > 1 rol | `PUT /usuarios/remove-role` (modal con select de rol a quitar) |

**Modales / diálogos:**
- **Modal aprobar:** Select de rol + botón "Aprobar".
- **Modal asignar rol:** Select de rol (roles disponibles de `GET /roles`) + botón "Asignar".
- **Modal quitar rol:** Select con los roles actuales del usuario + botón "Quitar".
- **Confirmación desactivar/activar:** Diálogo simple "¿Desactivar a [nombre]?" con Confirmar/Cancelar.
