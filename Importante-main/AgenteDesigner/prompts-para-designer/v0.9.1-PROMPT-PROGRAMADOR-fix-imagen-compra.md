# v0.9.1 — Fix: imagen de factura en Compras

**VERSIÓN:** v0.9.1
**SLUG:** fix-imagen-compra
**TIPO:** Bug fix — sin cambios de UI
**DEPENDENCIAS:** v0.9.0 (Proveedores)
**ESTIMACIÓN:** 15 min
**PRIORIDAD:** alta — bloqueante para subir imágenes de factura

---

## Descripción del bug

Al adjuntar la imagen de factura a una compra (`PUT /compras/{id}/imagen`), el backend
responde error porque espera un parámetro multipart llamado `file`.

**Causa raíz:** `src/api/api.js` crea la instancia axios con el header por defecto
`Content-Type: application/json`. Cuando se envía un `FormData`, axios debería setear
automáticamente `multipart/form-data` con el `boundary`, pero el header de instancia lo
sobreescribe y el backend no puede parsear la parte `file`.

El código de `http.js` intenta eliminarlo (`delete config.headers['Content-Type']`) pero
opera sobre `options.headers || {}` — un objeto vacío — que no es el header de la instancia
axios. Por eso la limpieza nunca efectivamente remueve el header problemático.

## PASO 1 — Corregir el interceptor en `src/api/api.js`

En el interceptor `api.interceptors.request.use`, **antes de agregar el token**,
detectar si `config.data` es una instancia de `FormData` y eliminar el `Content-Type`
del config para que axios lo derive correctamente a `multipart/form-data` con boundary.

```js
api.interceptors.request.use((config) => {
  // Para FormData: eliminar Content-Type para que axios setee multipart/form-data + boundary
  if (config.data instanceof FormData) {
    delete config.headers['Content-Type']
  }

  if (config.skipAuth) {
    delete config.skipAuth
    return config
  }
  const token = getToken()
  if (token) {
    config.headers.Authorization = `Bearer ${token}`
  }
  return config
})
```

**NO modificar** `http.js` ni `compras.js` — la lógica ahí ya es correcta (usa
`formData.append('file', file)` con la clave `file` que espera el backend).
