# v0.12.0 — Rediseño de vistas de detalle de stock (Drones, Baterías, Hélices, Docks, Antenas, Accesorios)

**VERSIÓN:** v0.12.0
**SLUG:** stock-detalle-redesign
**TIPO:** Prompt para AgenteDesigner — generá la spec en `design-specs/v0.12.0-stock-detalle-redesign.md`
**DEPENDENCIAS:** v0.11.0 (stock post-compra, incluye modal de edición)
**ESTIMACIÓN PROGRAMADOR:** 3-4h (luego de la spec)

---

## Pedido

La vista de detalle de cada tipo de equipo (ej. `/stock/drones/12`) es actualmente muy
básica: imagen fija, lista de campos en texto plano, y un botón de volver.
No es digna de un producto de calidad. El usuario quiere algo **moderno, estilizado
y con buena UX**. El resultado debe sentirse como un producto real y cuidado.

---

## Estado actual (qué hay hoy — el Designer debe leer esto como punto de partida)

```
LEER ANTES DE DISEÑAR:
- src/views/stock/StockDronDetalleView.vue       ← vista de referencia (todas son iguales en estructura)
- src/views/stock/StockDronesView.vue            ← vista de listado (para mantener coherencia visual)
- src/views/stock/StockView.vue                  ← hub de stock (tonos, estilo de cards)
- Importante-main/AgenteDesigner/prompts-para-designer/v0.11.0-PROMPT-PROGRAMADOR-stock-post-compra.md
  ← el programador ya va a agregar modal de edición en el detalle (leer para no duplicar)
```

**Problemas visuales del estado actual:**
1. Layout de una sola columna — imagen arriba, campos abajo. Sin jerarquía visual.
2. El estado del ítem (`NO_LLEGO`, `STOCK_ACTUAL`, etc.) no es prominente — se pierde entre los campos.
3. Los campos se muestran como pares label/value en texto plano — sin estructura ni agrupación.
4. No hay distinción entre campos primarios (nombre, marca, modelo) y secundarios (garantía, fecha).
5. Botones de acción sin contexto visual — el de edición que agrega v0.11.0 queda suelto al pie.
6. La imagen es siempre la misma foto genérica — no aporta valor real.
7. El breadcrumb es minimalista y no orienta al usuario dentro del sistema.

## Qué tipos de equipo usan esta misma vista

Todos los detalles comparten la **misma estructura de datos base** con variaciones menores:

| Tipo | Campos clave | Ruta |
|------|-------------|------|
| Dron | nombre, marca, modelo, nroSerie, estado, fechaCompra, garantia | `/stock/drones/:id` |
| Batería | ídem | `/stock/baterias/:id` |
| Hélice | ídem | `/stock/helices/:id` |
| Dock | ídem + posibles campos de conectividad | `/stock/docks/:id` |
| Antena RTK | ídem | `/stock/antenas-rtk/:id` |
| Antena Starlink | ídem | `/stock/antenas-starlink/:id` |
| Accesorio | nombre, tipo, cantidad, estado, descripción | `/stock/accesorios/:id` |

La spec debe diseñar **una plantilla** que funcione para todos. El programador la aplicará
a cada vista con mínimos cambios (TIPO, TITULO, IMAGEN de placeholder).

## Campos disponibles por ítem (del backend)

```
id              número interno
nombre          string (viene de descripcionEquipo si fue creado desde compra)
marca           string | null
modelo          string | null
numeroSerie     string | null
estado          enum: NO_LLEGO | STOCK_ACTUAL | EN_PROCESO | STOCK_ACTIVO | EN_MANTENIMIENTO | EN_DESUSO
fechaCompra     date string | null
garantia        string | null
[otros campos que el backend pueda devolver según tipo]
```

**Colores de estado definidos en el sistema (mantener consistencia):**
- `NO_LLEGO` → amarillo `#fef9c3 / #854d0e`
- `STOCK_ACTUAL` → azul `#dbeafe / #1e40af`
- `EN_PROCESO` → azul
- `STOCK_ACTIVO` → verde `#dcfce7 / #166534`
- `EN_MANTENIMIENTO` → naranja `#ffedd5 / #9a3412`
- `EN_DESUSO` → gris `#f1f5f9 / #475569`

## Qué quiere el usuario (objetivos de diseño)

1. **Status prominente** — el estado del ítem debe ser lo primero y más visible: badge grande,
   posiblemente con un color de acento en el encabezado o un indicador lateral.

2. **Hero section** — encabezado con nombre del ítem, tipo de equipo, estado y las acciones
   principales (Editar) en una sola franja, sin tener que scrollear.

3. **Agrupación de datos** — no lista plana de campos. Agrupar en secciones con título:
   - "Identificación" → nombre, marca, modelo, nro. de serie
   - "Estado y ubicación" → estado, fecha de compra, garantía
   - (Extensible para campos futuros por tipo)

4. **Imagen de placeholder inteligente** — si no hay foto real, mostrar un ícono/ilustración
   grande del tipo de equipo (dron, batería, hélice) en lugar de una foto que no aporta nada.
   O bien eliminar la imagen y compensar con un diseño más rico en datos.

5. **Acciones contextuales** — el botón "Editar" (o "Completar datos" cuando es NO_LLEGO)
   debe estar visible e integrado al hero, no al fondo de la página.

6. **Alerta NO_LLEGO** — cuando el estado es `NO_LLEGO`, mostrar un banner o callout
   destacado que indique "Este equipo está pendiente de llegada. Completá los datos al recibirlo."

7. **Navegación clara** — breadcrumb mejorado: `Stock / Drones / [nombre del ítem]`
   con links funcionales a cada nivel.

8. **Responsive** — funciona bien en tablet y móvil (el admin puede estar en campo).

## DISEÑO REQUERIDO

El Designer debe generar la spec completa en `design-specs/v0.12.0-stock-detalle-redesign.md`
cubriendo:

1. **Layout general** — describir estructura (ej. columna única con hero + secciones cards,
   o dos columnas en desktop). Definir max-width del contenedor.

2. **Hero section** — qué contiene, cómo se estructura visualmente el nombre + estado + acciones.
   Especificar si el badge de estado tiene un acento de color en el fondo del hero o es solo el chip.

3. **Secciones de datos** — qué campos van en cada tarjeta/sección, cómo se presentan los pares
   label/value (tipografía, jerarquía, grid de N columnas en desktop vs 1 en mobile).

4. **Banner NO_LLEGO** — dónde aparece (debajo del hero o dentro), qué icono usar (⚠️ o similar),
   qué texto exacto muestra, y si incluye CTA directo a editar.

5. **Placeholder de imagen** — definir si se mantiene (con mejor tratamiento) o se reemplaza
   por un ícono/ilustración. Si se mantiene, especificar aspect ratio, border-radius, tratamiento de fondo.

6. **Botones de acción** — posición en el hero (derecha del título), estilos, jerarquía
   (primario: Editar; secundario: Volver al listado).

7. **Breadcrumb** — formato, separador, niveles. ¿Link activo vs no-link en el nivel actual?

8. **Estados de la página** — loading (skeleton vs spinner), error (mensaje + reintentar),
   ítem no encontrado (404).

9. **Tokens de color y tipografía** — usar los del proyecto (`#0d7377` como acento principal,
   `#1e293b` para títulos, `#64748b` para labels, etc.) y definir nuevos si hacen falta.

10. **Criterios de aceptación** — checklist verificable para el programador.

## Scope excluido de esta versión

- Subida de foto real del equipo (no hay endpoint de imagen para equipos en esta versión del backend).
- Historial de cambios o auditoría del ítem.
- Vista de listado (`/stock/drones`) — esa vista queda para otra mejora.
- Dock, Antena RTK, Antena Starlink — si tienen campos específicos, dejarlo como "Supuesto abierto".
