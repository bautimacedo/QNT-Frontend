# v0.1.1 — Registro de usuario (pantalla pública)

**VERSIÓN:** v0.1.1  
**SLUG:** registro-usuario  
**DEPENDENCIAS:** v0.1.0 (auth, login, router, API client)  
**ESTIMACIÓN:** 1h–1.5h  
**PRIORIDAD:** alta (necesario antes de gestión de usuarios)

---

## Origen

> Pedido directo del responsable del proyecto. El backend ya expone `POST /auth/register`
> (ruta pública, sin token). El usuario registrado queda en estado `PENDIENTE_APROBACION`
> hasta que un ADMIN lo apruebe desde el panel de gestión.

---

## Descripción

Implementar la **pantalla de registro** como ruta pública (`/register`), accesible desde
la pantalla de Login. El formulario recoge nombre, apellido (opcional), email y contraseña
y hace `POST /auth/register`. Tras un registro exitoso, se muestra un mensaje de confirmación
indicando que la cuenta debe ser aprobada por un administrador, y se redirige al login.

### Contexto backend (INFORME_BACKEND_PARA_FRONTEND.md)

- **Endpoint:** `POST /api/qnt/v1/auth/register` — ruta pública, sin token.
- **Body (RegisterRequest — §6.4):**
  - `nombre` (string, obligatorio)
  - `apellido` (string, opcional)
  - `email` (string, obligatorio, formato email válido)
  - `password` (string, obligatorio, mín. 6 caracteres)
- **Respuestas:**
  - **201:** usuario creado (JSON). Estado = `PENDIENTE_APROBACION`, `activo: false`.
  - **400:** errores de validación (campos faltantes, email inválido, password < 6 chars).
  - **409:** email ya existe ("Ya existe un Usuario con email ...").
  - **500:** error interno.

### Pantalla y ruta

- **Ruta:** `/register` (pública, sin `requiresAuth`).
- **Relación con Login:** Desde el Login actual (`/login`) debe haber un link/botón tipo
  "¿No tenés cuenta? Registrate" que lleve a `/register`. Desde `/register` debe haber
  un link "Ya tengo cuenta — Iniciar sesión" que lleve a `/login`.
- **Estilo visual:** Coherente con `LoginView.vue` (misma estructura de página: sección de
  formulario + sección de branding/ilustración). Reutilizar estilos, componentes y layout
  existentes del Login para consistencia.

### Campos del formulario

| Campo | Tipo input | Obligatorio | Validación cliente |
|-------|-----------|-------------|-------------------|
| Nombre | text | Sí | No vacío |
| Apellido | text | No | — |
| Email | email | Sí | Formato email básico |
| Contraseña | password | Sí | Mín. 6 caracteres |
| Confirmar contraseña | password | Sí | Debe coincidir con Contraseña (solo frontend, el backend no recibe este campo) |

### Flujo principal

1. Usuario navega a `/register` (o desde link en Login).
2. Completa el formulario.
3. Click en "Crear cuenta" (o equivalente).
4. Validación cliente: campos obligatorios, email válido, password ≥ 6, passwords coinciden.
   - Si falla: mostrar errores inline bajo cada campo.
5. `POST /auth/register` con body `{ nombre, apellido, email, password }`.
6. **Éxito (201):** Mostrar mensaje de confirmación:
   > "Tu cuenta fue creada exitosamente. Un administrador debe aprobar tu acceso antes de que puedas iniciar sesión."
   - Botón "Ir a Iniciar sesión" que redirige a `/login`.
7. **Error 409:** Mostrar "Ya existe una cuenta con ese email."
8. **Error 400:** Mostrar mensaje(s) de validación del backend.
9. **Error genérico:** "Ocurrió un error. Intentá de nuevo."

### Estados de UI

| Estado | Cuándo | Comportamiento |
|--------|--------|----------------|
| Formulario | Al entrar a `/register` | Formulario vacío, botón habilitado. |
| Validando | Tras click en "Crear cuenta" | Spinner en botón, campos deshabilitados. |
| Error validación | Validación cliente falla | Errores inline, foco en primer campo con error. |
| Error API | 400/409/500 del backend | Mensaje de error visible (banner o inline según tipo). |
| Éxito | 201 | Ocultar formulario, mostrar mensaje de éxito + link a login. |

### Criterios de aceptación (para la spec)

- [ ] Ruta `/register` accesible sin autenticación.
- [ ] Si el usuario está autenticado y va a `/register`, redirigir a `/` (como ya se hace con `/login`).
- [ ] Formulario con 5 campos: nombre, apellido, email, contraseña, confirmar contraseña.
- [ ] Validación en cliente antes de enviar: obligatorios, formato email, password ≥ 6, passwords coinciden.
- [ ] `POST /auth/register` con body correcto (sin campo confirmar contraseña).
- [ ] Éxito: mensaje de cuenta pendiente de aprobación + link a login.
- [ ] Error 409: mensaje de email duplicado.
- [ ] Error 400: mensajes de validación del backend.
- [ ] Link desde Login → Registro y desde Registro → Login.
- [ ] Estilo visual coherente con LoginView.vue.
