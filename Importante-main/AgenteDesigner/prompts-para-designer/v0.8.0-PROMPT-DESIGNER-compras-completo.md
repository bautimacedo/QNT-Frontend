## 1. CONTEXTO DEL PROYECTO

### 1.1 Stack y convenciones

- **Framework:** Vue 3 (Composition API con `<script setup>`)

## 2. CONTRATO DEL BACKEND (resumen para esta feature)

### 2.1 Endpoints de Compras

| Método | Ruta | Descripción | Roles |
|---|---|---|---|
| GET | `/compras` | Listar todas | ADMIN, USER |
| GET | `/compras/{id}` | Obtener por ID | ADMIN, USER |
| POST | `/compras` | Crear compra | ADMIN, USER |
| PUT | `/compras/{id}` | Actualizar compra | ADMIN, USER |
| DELETE | `/compras/{id}` | Eliminar compra | **Solo ADMIN** |
| PUT | `/compras/{id}/imagen` | Subir imagen factura (multipart, parte `file`, máx 10MB) | ADMIN, USER |
| GET | `/compras/{id}/imagen` | Descargar imagen factura (binario) | ADMIN, USER |
| GET | `/compras/tipos-equipo` | Array de valores del enum TipoEquipo (para dropdown dinámico) | ADMIN, USER |

### 2.2 Modelo: Compra (respuesta del backend)

| Campo | Tipo | Notas |
|---|---|---|
| id | number | PK |
| proveedor | Proveedor (objeto) | `{ id, nombre, cuit, contacto, direccion, telefono, email, observaciones }` |
| fechaCompra | string (date) | ISO-8601 `yyyy-MM-dd`. **Obligatorio** |
| fechaFactura | string (date) \| null | |
| numeroFactura | string \| null | |
| importe | number | BigDecimal. **Obligatorio, > 0** |
| moneda | string | Default `"ARS"` |
| tipoCompra | string | Enum TipoCompra. **Obligatorio** |
| tipoEquipo | string \| null | Enum TipoEquipo. **Solo presente cuando tipoCompra = "EQUIPO"** |
| descripcionEquipo | string \| null | Texto libre, máx. 255 chars. **Solo aplica cuando tipoCompra = "EQUIPO"** |
| descripcion | string \| null | |
| site | Site \| null | `{ id, nombre }` |
| observaciones | string \| null | |
| imagenFactura | — | No se serializa; usar GET/PUT `.../imagen` |

### 2.3 Modelo: Proveedor (anidado en Compra)

| Campo | Tipo |
|---|---|
| id | number |
| nombre | string |
| cuit | string \| null |
| contacto | string \| null |
| direccion | string \| null |
| telefono | string \| null |
| email | string \| null |
| observaciones | string \| null |

**No hay CRUD dedicado de Proveedor.** Se crean desde Compras: si se envía `proveedorNombre` (sin `proveedorId`), el backend crea un proveedor nuevo automáticamente.

### 2.4 CreateCompraRequest (POST / PUT)

| Campo | Tipo | Obligatorio |
|---|---|---|
| proveedorId | number \| null | Uno de los dos |
| proveedorNombre | string \| null | Si no hay proveedorId, crea proveedor |
| fechaCompra | string (date) | **Sí** |
| fechaFactura | string (date) \| null | No |
| numeroFactura | string \| null | No |
| importe | number | **Sí, > 0** |
| moneda | string \| null | Default `"ARS"` |
| tipoCompra | string | **Sí** (enum TipoCompra) |
| tipoEquipo | string \| null | **Obligatorio si tipoCompra = "EQUIPO"** |
| descripcionEquipo | string \| null | Solo si tipoCompra = "EQUIPO" |
| descripcion | string \| null | No |
| siteId | number \| null | No |
| observaciones | string \| null | No |

### 2.5 Enums

**TipoCompra:**
`LICENCIA_SW`, `REPUESTO`, `COMBUSTIBLE`, `VIATICO`, `SEGURO`, `EQUIPO`, `OTRO`

**TipoEquipo** (dinámico, obtener de `GET /compras/tipos-equipo`):
`DRON`, `DOCK`, `BATERIA`, `HELICE`, `ANTENA_RTK`, `ANTENA_STARLINK`, `OTRO`

**Flujo condicional:**
- Si `tipoCompra == "EQUIPO"` → mostrar select de `tipoEquipo` (cargado dinámicamente) + input de `descripcionEquipo`
- Si `tipoCompra != "EQUIPO"` → ocultar ambos campos

### 2.6 Imagen de factura

- **Subir:** `PUT /compras/{id}/imagen` — `multipart/form-data`, parte `file`, máx. 10 MB
- **Obtener:** `GET /compras/{id}/imagen` — respuesta binaria (blob)
- **404** si no tiene imagen

### 2.7 Permisos

| Acción | ADMIN | USER |
|---|---|---|
| Listar compras | ✓ | ✓ |
| Ver detalle | ✓ | ✓ |
| Crear compra | ✓ | ✓ |
| Editar compra | ✓ | ✓ |
| **Eliminar compra** | **✓** | **✗** |
| Subir/ver imagen | ✓ | ✓ |

## 3. REQUERIMIENTOS FUNCIONALES

### 3.1 Vista principal: Listado de compras (`/compras`)

Una sola vista en ruta `/compras` que concentre todo el CRUD. Debe ser **la mejor vista del sistema** — profesional, limpia y funcional.

**Header de la página:**
- Título: "Compras"
- Botón "Nueva compra" (abre modal/drawer de creación)
- Indicador de cantidad total de compras

**Barra de filtros** (sobre la tabla):
- Filtro por texto libre (busca en proveedor, descripción, nº factura, observaciones) — filtrado client-side
- Filtro por Tipo de Compra (select con "Todos" + cada TipoCompra)
- Filtro por Moneda (select con "Todas" + las monedas presentes en los datos)
- Filtro por rango de fechas (fecha desde / fecha hasta) — filtra sobre `fechaCompra`
- Filtro por Proveedor (select con "Todos" + proveedores únicos extraídos de las compras)
- Botón/link "Limpiar filtros" cuando hay algún filtro activo
- Indicador: "Mostrando X de Y compras" debajo de los filtros

**Tabla de compras:**

| Columna | Campo | Formato |
|---|---|---|
| Proveedor | `proveedor.nombre` | Texto |
| Fecha | `fechaCompra` | dd/mm/yyyy |
| Nº Factura | `numeroFactura` | Texto o "—" |
| Tipo | `tipoCompra` | Badge con color por tipo (ver mapeo abajo) |
| Equipo | `tipoEquipo` | Mostrar solo si la compra es tipo EQUIPO; badge secundario |
| Importe | `importe` + `moneda` | Formato numérico con separador de miles + moneda (ej. "$12.500,00 ARS") |
| Acciones | — | Botones: Ver detalle, Editar, Imagen, Eliminar (solo ADMIN) |

**Mapeo de colores por TipoCompra (badges):**

| TipoCompra | Color badge | Label amigable |
|---|---|---|
| LICENCIA_SW | `#dbeafe` / `#1e40af` (azul) | Licencia SW |
| REPUESTO | `#fef3c7` / `#92400e` (amarillo) | Repuesto |
| COMBUSTIBLE | `#fee2e2` / `#991b1b` (rojo) | Combustible |
| VIATICO | `#e0e7ff` / `#3730a3` (índigo) | Viático |
| SEGURO | `#dcfce7` / `#166534` (verde) | Seguro |
| EQUIPO | `#f3e8ff` / `#6b21a8` (púrpura) | Equipo |
| OTRO | `#f1f5f9` / `#64748b` (gris) | Otro |

**Ordenamiento:**
- La tabla debe poder ordenarse haciendo click en los headers de columna (al menos: Fecha, Proveedor, Importe, Tipo)
- Default: por `fechaCompra` descendente (más reciente primero)

**Resumen/estadísticas** (card arriba de la tabla o cards laterales):
- **Total gastado** (suma de importes filtrados, agrupado por moneda)
- **Cantidad de compras** (filtradas)
- **Distribución por tipo** (opcional: mini-resumen de cuántas hay de cada tipo)

### 3.2 Modal: Crear / Editar compra

Un modal grande (`modal-card--wide` o incluso un side drawer) con formulario completo.

**Campos del formulario:**

| Campo UI | Campo API | Tipo input | Obligatorio | Notas |
|---|---|---|---|---|
| Proveedor | proveedorId / proveedorNombre | **Combo especial** (ver abajo) | Sí | |
| Fecha de compra | fechaCompra | date | Sí | |
| Fecha de factura | fechaFactura | date | No | |
| Nº Factura | numeroFactura | text | No | |
| Importe | importe | number (step="0.01") | Sí | Validar > 0 |
| Moneda | moneda | select | Sí | Opciones: ARS, USD, EUR. Default: ARS |
| Tipo de compra | tipoCompra | select | Sí | Opciones del enum TipoCompra con labels amigables |
| Tipo de equipo | tipoEquipo | select | **Sí si EQUIPO** | Solo visible si tipoCompra = "EQUIPO". Cargado de `GET /compras/tipos-equipo` |
| Descripción equipo | descripcionEquipo | text | No | Solo visible si tipoCompra = "EQUIPO". Máx. 255 chars |
| Descripción | descripcion | textarea | No | |
| Observaciones | observaciones | textarea | No | |

**Combo de Proveedor (UX clave):**
Debe permitir dos flujos:
1. **Seleccionar proveedor existente:** Dropdown/autocomplete que muestra los proveedores ya usados en compras anteriores (extraídos del listado de compras, deduplicados por `proveedor.id`). Mostrar `nombre` + `cuit` si tiene.
2. **Crear proveedor nuevo:** Si el usuario escribe un nombre que no existe, mostrar opción "Crear proveedor: [nombre ingresado]". En este caso, enviar `proveedorNombre` en vez de `proveedorId`.

**Validación client-side:**
- Proveedor: obligatorio (id o nombre)
- Fecha compra: obligatorio
- Importe: obligatorio, > 0
- Tipo compra: obligatorio
- Tipo equipo: obligatorio si tipo compra = "EQUIPO"

**Botones:**
- "Cancelar" (cierra sin guardar)
- "Guardar" (POST si crear, PUT si editar) → toast de éxito → cierra modal → recarga lista

### 3.3 Modal: Detalle de compra

Modal de solo lectura que muestra toda la información de una compra de forma presentable. Organizado en secciones:

**Sección 1: Información principal**
- Proveedor (nombre, cuit, contacto, teléfono, email, dirección si tiene)
- Tipo de compra (badge) + tipo equipo y descripción equipo si aplica
- Importe + moneda
- Descripción
- Observaciones

**Sección 2: Facturación**
- Fecha de compra
- Fecha de factura
- Nº de factura

**Sección 3: Imagen de factura**
- Si tiene imagen: mostrar preview (cargado desde `GET /compras/{id}/imagen`)
- Botón para subir/cambiar imagen
- Si no tiene: zona de upload con mensaje "No hay imagen de factura cargada"

**Acciones del detalle:**
- "Editar" → abre modal de edición con datos prellenados
- "Cerrar"

### 3.4 Imagen de factura

Puede estar dentro del modal de detalle o como acción separada:
- **Preview:** Mostrar la imagen como `<img>` con `object-fit: contain`, máximo 500px de ancho
- **Upload:** Zona de drag & drop o botón "Subir imagen" → `<input type="file" accept="image/*">` → `PUT /compras/{id}/imagen`
- **Reemplazar:** Si ya hay imagen, botón "Cambiar imagen"
- **Validación:** Máx. 10 MB, solo imágenes
- **Loading:** Spinner mientras sube/descarga

### 3.5 Eliminar compra (solo ADMIN)

- El botón "Eliminar" solo se muestra si el usuario tiene `ROLE_ADMIN`
- Click → modal de confirmación: "¿Eliminar la compra de [proveedor] del [fecha]?"
- Confirmar → `DELETE /compras/{id}` → toast "Compra eliminada" → recarga lista

### 3.6 Permisos en la UI

El frontend debe inyectar `dashboardUser` (provisto por `DashboardLayout`) para chequear roles:
- `ROLE_ADMIN` → ve botón Eliminar, acceso completo
- `ROLE_USER` → no ve botón Eliminar
- `ROLE_PILOTO` sin ROLE_USER ni ADMIN → según los permisos de backend (puede que no tenga acceso; en ese caso la vista podría no estar visible en el sidebar para ese rol, o mostrar "Sin permisos")

### 4.2 Experiencia de creación/edición

El modal de crear/editar debe sentirse **fluido y profesional**:
- Campos organizados en **2 columnas** donde tenga sentido (ej. fecha compra + fecha factura en la misma fila, importe + moneda en la misma fila)
- El combo de proveedor debe ser lo primero y más prominente
- Los campos condicionales (tipoEquipo, descripcionEquipo) deben aparecer con una transición suave cuando se selecciona "EQUIPO"
- Labels claros y consistentes
- Mensajes de error inline debajo de cada campo

### 4.3 Estados de la vista

| Estado | Cuándo | Cómo se muestra |
|---|---|---|
| Loading | Mientras `GET /compras` carga | Spinner centrado con "Cargando compras..." |
| Vacío | Lista vacía (0 compras en total) | Ilustración/ícono + "No hay compras registradas" + botón "Crear primera compra" |
| Sin resultados | Filtros activos pero 0 resultados | "No se encontraron compras con los filtros aplicados" + "Limpiar filtros" |
| Error | Fallo en la petición | Mensaje de error + "Reintentar" |
| Con datos | Lista cargada | Tabla con datos, filtros funcionales |

### 4.4 Responsive

- En pantallas < 768px: filtros apilados verticalmente, tabla con scroll horizontal
- Cards de resumen en 1 columna
- Modal ocupa casi todo el ancho de pantalla
- Formulario pasa a 1 columna

### 4.5 Formato de moneda

Mostrar importes con:
- Separador de miles: punto (.) para ARS, coma (,) para USD/EUR
- Separador decimal: coma (,) para ARS, punto (.) para USD/EUR
- Símbolo: $ para ARS, US$ para USD, € para EUR
- Ejemplo: `$12.500,00 ARS`, `US$1,200.00 USD`

O usar `Intl.NumberFormat('es-AR', { style: 'currency', currency: moneda })` para simplificar.

## 7. NOTAS PARA EL DESIGNER

1. **Proveedor sin CRUD propio:** No hay endpoint dedicado para gestionar proveedores. Los proveedores se crean indirectamente al crear una compra con `proveedorNombre`. Para el selector de proveedor, extraer los proveedores únicos del listado de compras (deduplicar por `proveedor.id`). Si en el futuro el backend agrega CRUD de proveedores, solo habría que cambiar la fuente del dropdown.

2. **Site:** No hay CRUD de Site en la API. El campo `siteId` existe en el request pero no hay endpoint para listar sites. **Decisión sugerida:** Omitir el campo Site del formulario por ahora (dejarlo para cuando exista el endpoint). Si querés incluirlo, hacerlo como campo de texto libre y documentar que es temporal.
