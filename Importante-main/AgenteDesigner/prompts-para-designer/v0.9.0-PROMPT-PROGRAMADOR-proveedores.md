## PASO 1 — Crear `src/api/proveedores.js`

Crear el archivo. Si el backend **ya expone** los endpoints, implementar las funciones reales. Si **no**, implementar funciones que permitan el Escenario B y dejen preparado el Escenario A.

### Cuando el backend SÍ expone `/proveedores` (Escenario A)

```js
import { request, json } from './http.js'

const BASE = '/proveedores'

export async function getProveedores() {
  const res = await request(BASE, { method: 'GET' })
  return json(res)
}

export async function getProveedor(id) {
  const res = await request(`${BASE}/${id}`, { method: 'GET' })
  return json(res)
}

export async function crearProveedor(body) {
  const res = await request(BASE, { method: 'POST', body })
  return json(res)
}

export async function actualizarProveedor(id, body) {
  const res = await request(`${BASE}/${id}`, { method: 'PUT', body })
  return json(res)
}

export async function eliminarProveedor(id) {
  const res = await request(`${BASE}/${id}`, { method: 'DELETE' })
  if (!res.ok) {
    const text = await res.text()
    throw new Error(text || 'Error al eliminar proveedor')
  }
  return null
}
```

### Cuando el backend NO expone `/proveedores` (Escenario B)

- **Opción 1:** Crear `proveedores.js` con funciones que usen `getCompras()` y extraigan proveedores únicos. Las funciones `crearProveedor`, `actualizarProveedor`, `eliminarProveedor` pueden lanzar un error claro: `throw new Error('El API de proveedores no está disponible. Los proveedores se muestran a partir de las compras.')` o simplemente no exportarlas y en la vista no mostrar botones de crear/editar/eliminar.
- **Opción 2:** Implementar `getProveedores()` como función que llama a `getCompras()` y devuelve `proveedoresUnicos` (deduplicar por `proveedor.id`). No implementar POST/PUT/DELETE hasta que exista el API.

Ejemplo de derivación desde compras:

```js
import { getCompras } from './compras.js'

export async function getProveedoresFromCompras() {
  const compras = await getCompras()
  const byId = new Map()
  compras.forEach(c => {
    if (c.proveedor && !byId.has(c.proveedor.id)) {
      byId.set(c.proveedor.id, { ...c.proveedor })
    }
  })
  return Array.from(byId.values()).sort((a, b) => (a.nombre || '').localeCompare(b.nombre || ''))
}
```

Decisión: en la vista, comprobar si existe un endpoint real (por ejemplo intentando `GET /proveedores` una vez y capturando 404) o usar una variable de configuración/entorno `VITE_PROVEEDORES_API=false` para elegir el modo. Por simplicidad, se puede implementar **solo Escenario B** primero (listado desde compras, sin CRUD) y documentar que cuando el backend agregue `/proveedores`, se añadan las funciones y se habiliten los botones.

## Notas para el programador

1. **Prioridad:** Si el backend no tiene `/proveedores`, implementar Escenario B primero (listado desde compras, sin CRUD). Cuando el backend agregue el CRUD, añadir las funciones en `proveedores.js` y habilitar los botones en la vista.
2. **Relación con Compras:** En ComprasView el selector de proveedor ya puede usar proveedores únicos de las compras. ProveedoresView puede reutilizar la misma lógica de extracción para el listado en B.
3. **Eliminar con compras:** Si el backend devuelve 400/409 con mensaje "No se puede eliminar porque tiene compras asociadas", mostrar ese mensaje en el modal de confirmación o en un toast de error.
4. **Link "Ver en Compras":** Navegar a `/compras?proveedorId=X`. La vista de Compras puede leer el query y prefiltrar por proveedor si ya implementaste ese filtro; si no, el link igualmente lleva a Compras y el usuario puede filtrar manualmente.
5. **CUIT:** Mostrar como viene del backend; no es obligatorio formatear (ej. 30-71234567-9) en esta versión.
