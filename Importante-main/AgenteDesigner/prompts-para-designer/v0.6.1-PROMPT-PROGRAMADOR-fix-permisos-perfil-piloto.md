# v0.6.1 — Fix: Perfil Piloto muestra "No tenés permisos" a un usuario con ROLE_PILOTO

**VERSIÓN:** v0.6.1  
**SLUG:** fix-permisos-perfil-piloto  
**DEPENDENCIAS:** v0.6.0  
**PRIORIDAD:** bloqueante — ningún piloto puede ver su perfil

---

## Descripción del bug

Al entrar a `/perfil-piloto` logueado como un usuario con ROLE_PILOTO, se muestra:

> "No tenés permisos de piloto para ver esta sección."

La vista debería mostrar las secciones de CMA, Licencias ANAC e Info de vuelo.

---

## Diagnóstico (causa raíz)

El check de permisos en `PerfilPilotoView.vue` es este (línea 45-48):

```js
const esPilotoOAdmin = computed(() => {
  if (!perfil.value?.roles) return false
  return perfil.value.roles.some(r => r.codigo === 'ROLE_PILOTO' || r.codigo === 'ROLE_ADMIN')
})
```

Esto depende de que `GET /mi-perfil` devuelva el campo `roles` como array de objetos con `codigo`. **El problema es que `GET /mi-perfil` probablemente no devuelve el campo `roles` en la respuesta (o lo devuelve null/vacío)**, haciendo que el computed siempre sea `false`.

Mientras tanto, el **DashboardLayout** ya tiene la info de roles del usuario logueado. Llama a `GET /auth/me` al montar, que devuelve:

```json
{ "id": 1, "email": "piloto@ejemplo.com", "username": "...", "authorities": ["ROLE_PILOTO"] }
```

Y lo provee como `dashboardUser` via `provide('dashboardUser', user)`.

**La vista `PerfilPilotoView.vue` no usa esa fuente.** Usa solo `perfil` (de `GET /mi-perfil`), que puede no traer roles.

## PASO 1 — Solución

Hacer que `esPilotoOAdmin` use **dos fuentes** para determinar el rol, priorizando la más confiable (`/auth/me` via `dashboardUser`):

### En `PerfilPilotoView.vue`, hacer estos cambios:

**1. Importar `inject`** (ya debería estar disponible desde vue):

```js
import { ref, computed, onMounted, onUnmounted, inject } from 'vue'
```

**2. Inyectar `dashboardUser`** (ya provisto por DashboardLayout):

```js
const dashboardUser = inject('dashboardUser', ref(null))
```

**3. Reescribir `esPilotoOAdmin`** para usar ambas fuentes:

```js
const esPilotoOAdmin = computed(() => {
  // Fuente 1: dashboardUser de /auth/me (tiene authorities: ["ROLE_PILOTO"])
  const authUser = dashboardUser.value
  if (authUser?.authorities?.length) {
    return authUser.authorities.some(a => a === 'ROLE_PILOTO' || a === 'ROLE_ADMIN')
  }

  // Fuente 2 (fallback): perfil de /mi-perfil (tiene roles: [{ codigo }])
  if (perfil.value?.roles?.length) {
    return perfil.value.roles.some(r => r.codigo === 'ROLE_PILOTO' || r.codigo === 'ROLE_ADMIN')
  }

  return false
})
```

**4. Ajustar el `onMounted`** para no depender del timing de `esPilotoOAdmin`:

Actualmente:
```js
onMounted(async () => {
  await loadPerfil()
  if (esPilotoOAdmin.value) { loadCma(); loadLicencias() }
})
```

El problema es que si `dashboardUser` aún no cargó cuando `loadPerfil()` termina, y `perfil` tampoco tiene roles, entonces `esPilotoOAdmin` es false y nunca carga CMA ni licencias.

Cambiar a:

```js
onMounted(async () => {
  await loadPerfil()
  loadPilotoDataIfAllowed()
})

function loadPilotoDataIfAllowed() {
  if (esPilotoOAdmin.value) {
    loadCma()
    loadLicencias()
  }
}
```

Y agregar un **watch** por si `dashboardUser` carga después de `perfil`:

```js
import { ref, computed, onMounted, onUnmounted, inject, watch } from 'vue'

// después de definir esPilotoOAdmin:
watch(esPilotoOAdmin, (newVal) => {
  // Si pasa a true y aún no cargamos CMA/licencias, cargarlos
  if (newVal && !cmaData.value && !licenciasLoading.value) {
    loadCma()
    loadLicencias()
  }
})
```

Esto cubre todos los escenarios de timing:
- Si `dashboardUser` ya cargó antes que `perfil` → funciona desde el `onMounted`.
- Si `dashboardUser` carga después → el `watch` lo detecta y carga CMA/licencias.
- Si `GET /mi-perfil` sí trae roles → funciona como fallback.

## Notas para el programador

1. **El fix es solo en `PerfilPilotoView.vue`.** No se necesitan cambios en API, router ni otros componentes.
2. **`dashboardUser` ya está provisto** por DashboardLayout via `provide('dashboardUser', user)`. Solo hay que hacer `inject('dashboardUser')`.
3. **`/auth/me` devuelve `authorities: ["ROLE_PILOTO"]`** (strings). **`/mi-perfil` devuelve `roles: [{ id, codigo, nombre }]`** (objetos). Son formatos distintos; el computed debe manejar ambos.
4. **Si querés verificar qué devuelve cada endpoint**, abrir las DevTools del navegador → Network → buscar las requests a `/auth/me` y `/mi-perfil` y comparar los JSON de respuesta. Confirmar si `/mi-perfil` trae o no el campo `roles`.
5. **Si confirmás que `/mi-perfil` SÍ devuelve `roles` correctamente**, entonces el bug es de timing (dashboardUser no cargó a tiempo). El watch lo resuelve igual.
