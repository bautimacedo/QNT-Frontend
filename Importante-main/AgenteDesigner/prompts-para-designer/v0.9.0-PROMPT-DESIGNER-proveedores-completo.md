## 0. SITUACIÓN DEL BACKEND

**Hoy:** El informe backend indica que **no hay CRUD dedicado de Proveedor**; los proveedores se crean/usan desde Compras (`proveedorId` o `proveedorNombre`).

**Para esta feature hay dos escenarios:**

| Escenario | Descripción | Qué implementar |
|---|---|---|
| **A) Backend ya expone CRUD Proveedores** | Existen `GET/POST/PUT/DELETE /proveedores` (y opcionalmente `GET /proveedores/{id}`) | Vista completa: listado, crear, editar, eliminar, filtros, detalle. |
| **B) Backend aún no expone CRUD** | Solo se puede obtener proveedores como parte de las compras (`GET /compras` → cada compra tiene `proveedor`) | Listado derivado: extraer proveedores únicos del listado de compras. Botón "Ver compras" por proveedor (filtrar compras por proveedor). Modales de Crear/Editar proveedor preparados pero deshabilitados o con mensaje "Disponible cuando el backend exponga el API de proveedores", o implementar creación vía "Nueva compra con este proveedor" como workaround. |

El diseño debe cubrir **el caso completo (A)**. Si el programador trabaja en (B), deberá adaptar: listado desde compras, sin POST/PUT/DELETE hasta que exista el API.

**Contrato de API esperado para Proveedores (cuando el backend lo implemente):**

| Método | Ruta | Descripción | Roles sugeridos |
|--------|------|-------------|------------------|
| GET | `/proveedores` | Listar todos los proveedores | ADMIN, USER |
| GET | `/proveedores/{id}` | Obtener un proveedor por ID | ADMIN, USER |
| POST | `/proveedores` | Crear proveedor (body: CreateProveedorRequest) | ADMIN, USER |
| PUT | `/proveedores/{id}` | Actualizar proveedor (body: UpdateProveedorRequest) | ADMIN, USER |
| DELETE | `/proveedores/{id}` | Eliminar proveedor (solo si no tiene compras asociadas, o según regla de negocio) | ADMIN |

**Modelo Proveedor (ya definido en informe §6.13):**

| Campo | Tipo | Notas |
|-------|------|--------|
| id | number | PK |
| nombre | string | Obligatorio |
| cuit | string \| null | |
| contacto | string \| null | |
| direccion | string \| null | |
| telefono | string \| null | |
| email | string \| null | |
| observaciones | string \| null | |

**Request esperado para crear/actualizar (CreateProveedorRequest / UpdateProveedorRequest):**

| Campo | Tipo | Obligatorio |
|-------|------|-------------|
| nombre | string | Sí (crear); Sí (editar) |
| cuit | string \| null | No |
| contacto | string \| null | No |
| direccion | string \| null | No |
| telefono | string \| null | No |

## 2. REQUERIMIENTOS FUNCIONALES

### 2.1 Vista principal: Listado de proveedores (`/proveedores`)

Una sola vista en la ruta `/proveedores` que reemplace el placeholder actual.

**Header de la página:**
- Título: "Proveedores"
- Botón "Nuevo proveedor" (abre modal de creación; si API no existe, puede estar deshabilitado o abrir un mensaje informativo)
- Indicador de cantidad total de proveedores

**Barra de filtros** (sobre la tabla):
- **Búsqueda por texto libre:** busca en nombre, CUIT, contacto, email, observaciones (filtrado client-side).
- **Filtro por “tiene compras”:** opcional — select "Todos" / "Con compras" / "Sin compras" (si hay API de proveedores y se puede saber si tiene compras; si no, omitir o derivar desde compras).
- Botón/link **"Limpiar filtros"** cuando hay algún filtro activo.
- Texto: **"Mostrando X de Y proveedores"**.

**Tabla de proveedores:**

| Columna | Campo | Formato |
|---------|--------|---------|
| Nombre | nombre | Texto, puede ser link al detalle |
| CUIT | cuit | Texto o "—" si vacío |
| Contacto | contacto | Texto o "—" |
| Teléfono | telefono | Texto o "—" |
| Email | email | Texto o "—"; si tiene valor, mostrarlo como mailto o texto |
| Compras | (derivado) | Número de compras asociadas (si se puede calcular desde el listado de compras); o "—" |
| Acciones | — | Ver detalle, Editar, Eliminar (Eliminar solo ADMIN si el backend lo restringe) |

**Ordenamiento:**
- Ordenar por columnas haciendo click en el header (al menos: Nombre, CUIT, Contacto).
- Orden por defecto: **nombre** ascendente (A–Z).

**Resumen (opcional):**
- Card con "Total de proveedores".
- Si hay datos de compras: "Proveedores con al menos una compra: X".

### 2.2 Modal: Crear / Editar proveedor

Modal con formulario completo. Solo activo si el backend expone POST/PUT `/proveedores`.

**Campos del formulario:**

| Campo UI | Campo API | Tipo input | Obligatorio |
|----------|-----------|------------|-------------|
| Nombre | nombre | text | Sí |
| CUIT | cuit | text | No |
| Contacto | contacto | text | No |
| Dirección | direccion | text o textarea | No |
| Teléfono | telefono | text | No |
| Email | email | email | No |
| Observaciones | observaciones | textarea | No |

**Validación client-side:**
- Nombre: obligatorio, no vacío.
- Email: si se completa, formato email válido.

**Botones:**
- "Cancelar" (cierra sin guardar).
- "Guardar" (POST si crear, PUT si editar) → toast de éxito → cierra modal → recarga listado.

### 2.3 Modal o panel: Detalle de proveedor

Vista de solo lectura con toda la información del proveedor.

**Secciones:**
1. **Datos principales:** Nombre, CUIT, Contacto, Teléfono, Email, Dirección, Observaciones.
2. **Compras asociadas (opcional):** Lista de compras donde figura este proveedor (fecha, nº factura, importe, tipo). Si el listado viene de compras, filtrar por `proveedor.id`. Botón "Ver en Compras" que lleve a `/compras?proveedorId=X` si existe ese filtro en la vista de compras.

**Acciones en detalle:**
- "Editar" → abre modal de edición con datos prellenados.
- "Cerrar".

### 2.4 Eliminar proveedor

- Botón "Eliminar" en la fila (y/o en el detalle).
- Solo visible para **ADMIN** si el backend restringe DELETE a ADMIN.
- Al hacer click → **modal de confirmación:** "¿Eliminar el proveedor [nombre]? Esta acción no se puede deshacer."
- Si el backend no permite eliminar cuando tiene compras, mostrar mensaje del backend o genérico: "No se puede eliminar porque tiene compras asociadas."
- Confirmar → DELETE → toast "Proveedor eliminado" → recarga listado.

### 2.5 Permisos en la UI

- Usar `dashboardUser` (inyectado por `DashboardLayout`) para roles.
- **ROLE_ADMIN:** ver Eliminar, acceso completo.
- **ROLE_USER:** sin Eliminar (si el backend lo restringe).
- Si no hay API de proveedores: ocultar o deshabilitar "Nuevo proveedor", "Editar", "Eliminar", y mostrar listado derivado de compras con solo "Ver detalle" y "Ver compras".

## 3. FUENTE DE DATOS SEGÚN ESCENARIO

### Escenario A — Backend expone CRUD

- **Listado:** `GET /proveedores`.
- **Detalle:** `GET /proveedores/{id}`.
- **Crear:** `POST /proveedores` con body CreateProveedorRequest.
- **Actualizar:** `PUT /proveedores/{id}` con body UpdateProveedorRequest.
- **Eliminar:** `DELETE /proveedores/{id}`.
- **Compras por proveedor:** seguir usando `GET /compras` y filtrar en cliente por `compra.proveedor.id === id`, o un futuro endpoint `GET /compras?proveedorId=` si existe.

### Escenario B — Sin API de proveedores

- **Listado:** `GET /compras` → extraer proveedores únicos por `proveedor.id` (y nombre para ordenar). Mostrar en la misma tabla.
- **Detalle:** mismo objeto proveedor; "Compras asociadas" = compras donde `compra.proveedor.id === proveedor.id`.
- **Crear/Editar/Eliminar:** no disponibles; botones ocultos o deshabilitados con tooltip/mensaje: "Próximamente" o "Requiere API de proveedores".

El diseño debe contemplar ambos escenarios para que la UI sea coherente cuando se agregue el API.

### 4.2 Estados de la vista

|--------|--------|-------------------|

- En pantallas &lt; 768px: filtros apilados, tabla con scroll horizontal.
- Modal en ancho completo o casi.

- Misma familia de fuentes, colores, radios, espaciados.
- Mismos patrones de modal, toast, confirmación y tabla.

## 6. CRITERIOS DE ACEPTACIÓN

### Listado y filtros
- [ ] La vista `/proveedores` muestra un listado de proveedores (desde API o derivado de compras).
- [ ] Filtro por texto libre busca en nombre, CUIT, contacto, email, observaciones.
- [ ] "Limpiar filtros" resetea todos los filtros.
- [ ] "Mostrando X de Y proveedores" es correcto.
- [ ] Ordenamiento por columnas (al menos Nombre) funciona.
- [ ] Orden por defecto es por nombre A–Z.

### CRUD (cuando el backend lo permita)
- [ ] Se puede crear un proveedor con todos los campos (POST).
- [ ] Se puede editar un proveedor (PUT).
- [ ] Solo ADMIN puede eliminar (si aplica); el botón no se muestra a USER.
- [ ] Modal de confirmación antes de eliminar.
- [ ] Toast en éxito al crear, editar y eliminar.

### Detalle
- [ ] Se puede abrir el detalle de un proveedor (modal o panel).
- [ ] Se muestran todos los campos del proveedor.
- [ ] Opcional: se listan las compras asociadas y/o enlace a Compras filtrado por proveedor.

### Estados y permisos
- [ ] Loading con spinner.
- [ ] Estado vacío con mensaje y CTA adecuado.
- [ ] Estado error con Reintentar.
- [ ] Sin resultados con "Limpiar filtros".
- [ ] Permisos según rol (Eliminar solo ADMIN si aplica).

### Escenario sin API
- [ ] Si no hay endpoints de proveedores, el listado se construye a partir de proveedores únicos de las compras.
- [ ] Crear/Editar/Eliminar deshabilitados o con mensaje claro.
- [ ] Detalle y "Ver compras" funcionan con los datos disponibles.

### Calidad
- [ ] Build sin errores.
- [ ] Sin errores en consola.
- [ ] Responsive.
- [ ] Textos en español.
- [ ] Estilos alineados al resto de la app.

## 7. NOTAS PARA EL DESIGNER

1. **Prioridad del diseño:** Pensar primero el caso con API completo (A); luego describir en la spec cómo se adapta al caso sin API (B), para que el programador pueda implementar ambos o solo uno.
2. **Relación con Compras:** En la vista de Compras, el selector de proveedor puede alimentarse de este listado (GET proveedores o lista única desde compras). Conviene que el modelo y la forma de mostrar proveedor sean coherentes entre ambas vistas.
3. **CUIT:** Puede mostrarse formateado (ej. 30-71234567-9) si el backend lo envía así; si no, mostrarlo como texto plano.
4. **Eliminar con compras:** Si el backend no permite eliminar un proveedor con compras, el mensaje de error debe mostrarse de forma clara en el modal o en un toast.
