## 1. Alcance y usuarios

**Qué se diseña:** Pantalla de registro público (`/register`) con formulario de alta de usuario, validación en cliente, integración con `POST /auth/register` y flujo de confirmación post-registro. Incluye también el ajuste del LoginView existente para vincular ambos flujos (Login ↔ Registro).

**Usuario(s) objetivo:** Cualquier persona sin cuenta que desee registrarse en el sistema. No requiere autenticación.

**Problema que resuelve:** Actualmente solo un ADMIN puede crear usuarios. Esta pantalla permite que cualquier persona se auto-registre; su cuenta queda en estado `PENDIENTE_APROBACION` hasta que un ADMIN la apruebe.

**Scope excluido (fuera de esta spec):** Panel de aprobación de usuarios pendientes (será otra versión). No se implementa flujo de "olvidé mi contraseña" ni verificación de email.

## 4. Estados y datos

**Origen de datos:** API REST — `POST /api/qnt/v1/auth/register` (ruta pública, sin token).

**Modelo de request (RegisterRequest — §6.4):**

| Campo | Tipo | Obligatorio | Validación cliente |
|-------|------|-------------|-------------------|
| nombre | string | Sí | No vacío, trim |
| apellido | string | No | — |
| email | string | Sí | Formato email básico (regex o `type="email"`) |
| password | string | Sí | Mín. 6 caracteres |

Campo adicional solo en frontend:
| Campo | Tipo | Obligatorio | Validación cliente |
|-------|------|-------------|-------------------|
| confirmPassword | string | Sí | Debe coincidir exactamente con `password` |

**Estados de UI a contemplar:**

| Estado | Cuándo | Comportamiento / mensaje |
|--------|--------|---------------------------|
| Formulario inicial | Al entrar a `/register` | Formulario vacío, todos los campos habilitados, botón "Crear cuenta" habilitado. |
| Validación cliente fallida | Tras click en "Crear cuenta" si hay errores | Errores inline bajo cada campo con error. Foco automático en el primer campo con error. Botón sigue habilitado. |
| Enviando (loading) | Tras pasar validación, mientras POST en curso | Spinner en el botón (texto "Creando cuenta…"), todos los campos y el botón deshabilitados. |
| Error 409 (email duplicado) | Backend devuelve 409 | Banner de error sobre el botón: "Ya existe una cuenta con ese email." Campos habilitados para corregir. |
| Error 400 (validación backend) | Backend devuelve 400 | Mostrar mensaje(s) del cuerpo de la respuesta en banner de error. |
| Error genérico (500 / red) | Fallo de red o 500 | Banner de error: "Ocurrió un error. Intentá de nuevo." |
| Éxito (201) | Backend devuelve 201 Created | Ocultar formulario completo. Mostrar panel de éxito con mensaje de confirmación y botón para ir a login. |

## 5. Flujos de usuario

**Flujo principal:**

1. Usuario navega a `/register` (directamente o desde link en Login).
2. Ve formulario con 5 campos vacíos y sección de branding a la derecha.
3. Completa nombre, email, contraseña y confirmar contraseña (apellido opcional).
4. Click en "Crear cuenta".
5. **Validación cliente:**
   - `nombre` no vacío → error: "El nombre es obligatorio."
   - `email` no vacío y formato válido → error: "Ingresá un email válido."
   - `password` ≥ 6 caracteres → error: "La contraseña debe tener al menos 6 caracteres."
   - `confirmPassword` === `password` → error: "Las contraseñas no coinciden."
   - Si alguna falla: mostrar errores inline, poner foco en el primer campo con error, NO enviar al backend.
6. Si validación pasa: deshabilitar campos, spinner en botón, `POST /auth/register` con body `{ nombre, apellido, email, password }` (sin `confirmPassword`).
7. **Respuesta 201:** Transicionar a estado de éxito. Mostrar:
   > "Tu cuenta fue creada exitosamente. Un administrador debe aprobar tu acceso antes de que puedas iniciar sesión."
   
   Botón "Ir a Iniciar sesión" que navega a `/login`.
8. **Respuesta 409:** Mostrar "Ya existe una cuenta con ese email." en banner. Habilitar campos.
9. **Respuesta 400:** Mostrar mensaje del cuerpo en banner. Habilitar campos.
10. **Error de red / 500:** Mostrar "Ocurrió un error. Intentá de nuevo." en banner. Habilitar campos.

**Flujos secundarios:**

- Desde Login → Registro: usuario está en `/login`, hace click en "Regístrate" (botón en sección derecha o link debajo del formulario) → navega a `/register`.
- Desde Registro → Login: usuario está en `/register`, hace click en "Ya tengo cuenta — Iniciar sesión" (debajo del formulario) → navega a `/login`.
- Usuario autenticado visita `/register`: el guard del router redirige a `/` (misma lógica que con `/login`).

**Eventos clave:**

| Evento | Acción |
|--------|--------|
| `@submit.prevent` del form | Ejecutar validación → si OK, POST /auth/register |
| Click "Ir a Iniciar sesión" (éxito) | `router.push('/login')` |
| Click "Ya tengo cuenta" (link) | `router-link to="/login"` |

**Validación — detalle de mensajes de error inline:**

| Campo | Condición de error | Mensaje |
|-------|-------------------|---------|
| nombre | Vacío (tras trim) | "El nombre es obligatorio." |
| email | Vacío | "El email es obligatorio." |
| email | Formato inválido | "Ingresá un email válido." |
| password | Vacío | "La contraseña es obligatoria." |
| password | < 6 caracteres | "La contraseña debe tener al menos 6 caracteres." |
| confirmPassword | Vacío | "Confirmá tu contraseña." |
| confirmPassword | No coincide con password | "Las contraseñas no coinciden." |

Los errores se muestran como `<p class="field-error">` debajo del input correspondiente (color `#dc2626`, font-size `0.85rem`).

## 6. Criterios de aceptación

- [ ] Ruta `/register` existe y es accesible sin autenticación.
- [ ] Si el usuario está autenticado y navega a `/register`, se redirige a `/`.
- [ ] La pantalla tiene layout 50/50 idéntico al Login (formulario izquierda, branding derecha).
- [ ] Formulario tiene 5 campos: Nombre, Apellido, Email, Contraseña, Confirmar Contraseña.
- [ ] Campo Apellido es opcional; el resto son obligatorios.
- [ ] Validación en cliente antes de enviar: nombre no vacío, email válido, password ≥ 6, passwords coinciden.
- [ ] Errores de validación se muestran inline debajo de cada campo con error.
- [ ] Al validar, el foco se mueve al primer campo con error.
- [ ] Durante envío: spinner en botón con texto "Creando cuenta…", campos deshabilitados.
- [ ] `POST /auth/register` envía body `{ nombre, apellido, email, password }` (sin `confirmPassword`).
- [ ] Éxito (201): se oculta el formulario y se muestra mensaje de cuenta pendiente de aprobación con botón "Ir a Iniciar sesión".
- [ ] Error 409: se muestra "Ya existe una cuenta con ese email." en banner visible.
- [ ] Error 400: se muestra el mensaje de validación del backend.
- [ ] Error genérico (500 / red): se muestra "Ocurrió un error. Intentá de nuevo."
- [ ] Link desde Login → Registro: botón "Regístrate" en sección de branding apunta a `/register`.
- [ ] Link desde Login → Registro: link de texto debajo del formulario de login.
- [ ] Link desde Registro → Login: "Ya tengo cuenta — Iniciar sesión" debajo del formulario de registro.
- [ ] Estilos coherentes con `LoginView.vue` (misma paleta, card, inputs, botón primario, tipografía).
- [ ] Responsive: en móvil (≤768px) la sección de branding va arriba, el formulario abajo (como LoginView).
