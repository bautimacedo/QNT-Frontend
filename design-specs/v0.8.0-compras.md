## 1. Alcance y usuarios

**Qué se diseña:** Vista completa de gestión de compras en `/compras` con: listado con filtros avanzados y ordenamiento, cards de resumen estadístico, modal de crear/editar compra (con selector inteligente de proveedor y campos condicionales para tipo EQUIPO), modal de detalle con imagen de factura, eliminación con confirmación (solo ADMIN). Esta será la vista más completa y pulida del sistema.

**Usuario(s) objetivo:**
- **ROLE_ADMIN:** CRUD completo incluyendo eliminar compras.
- **ROLE_USER:** Listar, crear, editar compras y gestionar imágenes de factura. No puede eliminar.

**Problema que resuelve:** Centralizar la gestión de compras operativas (licencias, repuestos, combustible, equipos, seguros, viáticos) en una interfaz profesional con búsqueda, filtrado, estadísticas y gestión de facturas.

**Scope excluido (fuera de esta spec):**
- CRUD dedicado de Proveedores (no existe endpoint; se crean indirectamente al crear compra con `proveedorNombre`).
- CRUD de Sites (no hay endpoint `GET /sites`; se omite `siteId` del formulario).
- Paginación server-side (los listados no están paginados en el backend).

### 3.3 Combo de Proveedor (UX clave)

**Comportamiento:**

1. Input de texto con **autocompletado**. Al escribir, se filtra la lista de proveedores conocidos (extraídos del listado de compras, deduplicados por `proveedor.id`).
2. Si hay coincidencias, se muestra un dropdown con los proveedores que coinciden. Cada opción muestra `nombre` + `cuit` (si tiene).
3. Al seleccionar un proveedor, se guarda `proveedorId` internamente.
4. Si el texto no coincide con ningún proveedor existente, se muestra una última opción: **"+ Crear proveedor: [texto ingresado]"**. Al seleccionarla, se guarda `proveedorNombre` (el backend crea el proveedor automáticamente).
5. Campo obligatorio: debe haber un `proveedorId` o un `proveedorNombre` al enviar.

**Implementación sugerida:** Input controlado + dropdown posicionado absolutamente. No es un `<select>` nativo sino un custom dropdown. Al escribir se filtra; al hacer click en una opción se selecciona; al perder foco se cierra el dropdown.

## 4. Estados y datos

### 4.1 Origen de datos

**API REST:**
| Endpoint | Uso |
|----------|-----|
| `GET /compras` | Listado completo al montar la vista |
| `GET /compras/{id}` | Detalle individual (para recargar tras edición) |
| `POST /compras` | Crear compra nueva |
| `PUT /compras/{id}` | Actualizar compra existente |
| `DELETE /compras/{id}` | Eliminar compra (solo ADMIN) |
| `PUT /compras/{id}/imagen` | Subir imagen factura (multipart, parte `file`, máx 10MB) |
| `GET /compras/{id}/imagen` | Descargar imagen factura (respuesta blob binario) |
| `GET /compras/tipos-equipo` | Array de strings TipoEquipo (para dropdown dinámico) |

**Datos derivados (client-side, no del backend):**
- Proveedores únicos: extraídos de `compras.map(c => c.proveedor)`, deduplicados por `id`.
- Monedas presentes: extraídas de `compras.map(c => c.moneda)`, deduplicadas.
- Resúmenes: totales, cantidad y distribución por tipo — calculados sobre la lista filtrada.

**Permisos (de `inject('dashboardUser')`):**
- `isAdmin`: `user.authorities?.includes('ROLE_ADMIN')`
- Controla: visibilidad del botón Eliminar.

### 4.3 Formato de moneda

Usar `Intl.NumberFormat` de JavaScript:

```js
function formatCurrency(amount, currency = 'ARS') {
  const locale = currency === 'ARS' ? 'es-AR' : 'en-US'
  return new Intl.NumberFormat(locale, {
    style: 'currency',
    currency,
    minimumFractionDigits: 2,
  }).format(amount)
}
```

Resultado: `$12.500,00` (ARS), `US$1,200.00` (USD), `€950.00` (EUR).

### 4.4 Formato de fechas

ISO `yyyy-MM-dd` del backend → mostrar como `dd/mm/yyyy` en la tabla y detalle:

```js
function formatDate(isoDate) {
  if (!isoDate) return '—'
  const [y, m, d] = isoDate.split('-')
  return `${d}/${m}/${y}`
}
```

## 5. Flujos de usuario

### 5.1 Flujo principal: Listar y filtrar compras

1. Usuario navega a `/compras`.
2. `GET /compras` se ejecuta al montar → loading → datos cargados.
3. Se calculan: proveedores únicos, monedas presentes, resúmenes.
4. Se muestra: cards de resumen, filtros (todos en valor por defecto), tabla ordenada por fecha desc.
5. Usuario aplica filtros → la lista se filtra client-side (AND lógico).
6. El indicador "Mostrando X de Y" se actualiza. Los resúmenes se recalculan sobre la lista filtrada.
7. "Limpiar filtros" → resetea todos a su valor por defecto.

### 5.2 Flujo: Crear compra

1. Click en "+ Nueva compra" → abre modal.
2. También se llama a `GET /compras/tipos-equipo` (si no se cargó antes) para tener los tipos disponibles.
3. Usuario completa los campos. Al seleccionar tipoCompra = "EQUIPO", aparecen `tipoEquipo` y `descripcionEquipo` con transición suave.
4. Al escribir en Proveedor, se filtra la lista de proveedores conocidos en un dropdown. Si no hay coincidencia, se muestra "+ Crear proveedor: [texto]".
5. Click en "Guardar" → validación client-side.
6. Si errores → mensajes inline bajo cada campo, foco en el primero.
7. Si OK → `POST /compras` con body `CreateCompraRequest`. Spinner en botón, campos deshabilitados.
8. Éxito (201) → cerrar modal → toast "Compra creada exitosamente" → recargar lista `GET /compras`.
9. Error 400 → mostrar mensaje en banner dentro del modal.
10. Error 500/red → "Ocurrió un error. Intentá de nuevo."

### 5.3 Flujo: Editar compra

1. Click en "Editar" en una fila → abre modal con datos precargados del la compra seleccionada.
2. Proveedor: se muestra el nombre actual. Si se cambia, mismo comportamiento que en crear.
3. Click en "Guardar" → `PUT /compras/{id}` → toast "Compra actualizada" → recargar lista.

### 5.4 Flujo: Ver detalle

1. Click en "Ver" en una fila → abre modal de detalle.
2. Se muestran todas las secciones (info, facturación, imagen).
3. Si la compra podría tener imagen, se intenta `GET /compras/{id}/imagen`. Si 404 → "No hay imagen".
4. Desde el detalle, botón "Editar" → cierra detalle → abre modal de edición.

### 5.5 Flujo: Imagen de factura

**Desde el modal de detalle:**
1. Si no hay imagen: botón "Subir imagen" → `<input type="file" accept="image/*">` oculto → seleccionar → validar < 10MB → `PUT /compras/{id}/imagen` (FormData, parte `file`) → spinner → éxito → recargar imagen → toast.
2. Si hay imagen: se muestra preview + botón "Cambiar imagen" → mismo flujo de upload.
3. Validación frontend: si `file.size > 10 * 1024 * 1024` → "La imagen no puede superar 10 MB" (no enviar).

### 5.6 Flujo: Eliminar compra (solo ADMIN)

1. Botón "Eliminar" visible solo si `isAdmin`.
2. Click → modal de confirmación: "¿Eliminar la compra de [proveedor.nombre] del [fechaCompra formateada]?"
3. Confirmar → `DELETE /compras/{id}` → spinner en botón → éxito (204) → toast "Compra eliminada" → recargar lista.
4. Error → toast con mensaje de error.

### 5.7 Flujo: Ordenamiento

1. Click en header de columna (Proveedor, Fecha, Importe, Tipo) → ordenar ASC.
2. Click de nuevo → DESC.
3. Indicador visual: flecha ↑/↓ en el header activo.
4. Default al cargar: `fechaCompra` DESC (más reciente primero).

### Validación — Crear/Editar (mensajes de error inline)

| Campo | Condición de error | Mensaje |
|-------|-------------------|---------|
| Proveedor | Ni seleccionado ni nombre ingresado | "Seleccioná o ingresá un proveedor." |
| Fecha de compra | Vacío | "La fecha de compra es obligatoria." |
| Importe | Vacío o ≤ 0 | "El importe debe ser mayor a 0." |
| Tipo de compra | No seleccionado | "Seleccioná un tipo de compra." |
| Tipo de equipo | Vacío cuando tipoCompra = "EQUIPO" | "Seleccioná un tipo de equipo." |

## 6. Criterios de aceptación

### Funcionalidad core
- [ ] La vista `/compras` muestra un listado completo cargado desde `GET /compras`.
- [ ] Se puede crear una nueva compra con todos los campos (`POST /compras`).
- [ ] Se puede editar una compra existente (`PUT /compras/{id}`).
- [ ] Solo ADMIN puede eliminar compras (`DELETE /compras/{id}`); el botón no se muestra para USER.
- [ ] Al crear compra con proveedor nuevo, se envía `proveedorNombre` y el backend lo crea.
- [ ] Al crear compra con proveedor existente, se envía `proveedorId`.

### TipoEquipo condicional
- [ ] Al seleccionar tipoCompra = "EQUIPO", aparecen los campos tipoEquipo y descripcionEquipo con transición suave.
- [ ] Al seleccionar otro tipoCompra, se ocultan esos campos y se limpian sus valores.
- [ ] tipoEquipo se carga dinámicamente desde `GET /compras/tipos-equipo`.
- [ ] tipoEquipo es obligatorio cuando tipoCompra = "EQUIPO".

### Filtros
- [ ] Filtro por texto libre funciona sobre proveedor, descripción, nº factura y observaciones.
- [ ] Filtro por TipoCompra filtra correctamente.
- [ ] Filtro por moneda filtra correctamente.
- [ ] Filtro por proveedor filtra correctamente.
- [ ] Filtro por rango de fechas (desde/hasta) funciona sobre `fechaCompra`.
- [ ] Los filtros se combinan (AND lógico).
- [ ] "Limpiar filtros" resetea todos los filtros.
- [ ] El indicador "Mostrando X de Y compras" es correcto y se actualiza dinámicamente.

### Ordenamiento
- [ ] Se puede ordenar por columna clickeando en el header (Proveedor, Fecha, Importe, Tipo).
- [ ] El ordenamiento default es por fecha descendente.
- [ ] Indicador visual de dirección de ordenamiento.

### Imagen de factura
- [ ] Se puede subir una imagen de factura (`PUT /compras/{id}/imagen`).
- [ ] Se puede ver la imagen cargada (`GET /compras/{id}/imagen`, respuesta blob → objectURL).
- [ ] Se puede reemplazar la imagen.
- [ ] Se valida tamaño máximo de 10 MB antes de enviar.
- [ ] Se muestra spinner mientras carga/sube.
- [ ] Se limpia objectURL al cerrar modal (`URL.revokeObjectURL`).

### Resumen estadístico
- [ ] Se muestra el total gastado (suma de compras filtradas) agrupado por moneda.
- [ ] Se muestra la cantidad de compras filtradas.
- [ ] Se muestra el tipo de compra más frecuente (de las filtradas).
- [ ] Los resúmenes se recalculan al cambiar filtros.

### Estados
- [ ] Estado loading con spinner.
- [ ] Estado vacío con CTA "Crear primera compra".
- [ ] Estado error con botón Reintentar.
- [ ] Estado sin resultados de filtros con "Limpiar filtros".
- [ ] Toast en acciones exitosas (crear, editar, eliminar, subir imagen).
- [ ] Modal de confirmación antes de eliminar.

### Validación
- [ ] Campos obligatorios marcados con `*` y validados antes de enviar.
- [ ] Errores inline debajo de cada campo con error.
- [ ] Errores del backend (400, 409, 500) se muestran como banner en el modal.
- [ ] No se puede enviar el formulario mientras hay un request en curso.

### Calidad
- [ ] Build sin errores (`npm run build`).
- [ ] Sin errores en consola del navegador.
- [ ] Responsive: en pantallas < 768px filtros apilados, tabla con scroll horizontal, modal ancho completo, formulario 1 columna.
- [ ] Todos los textos en español.
- [ ] Estilos 100% consistentes con UsuariosView (mismos tokens, tipografía, badges, modales, tabla, toast, spinner).

### Supuestos y abiertos

1. **Proveedor sin CRUD propio:** Los proveedores se obtienen del listado de compras (deduplicados por `proveedor.id`). Si la lista de compras está vacía, el usuario solo puede crear proveedores nuevos escribiendo el nombre.
2. **Site omitido:** No hay `GET /sites`. Se omite `siteId` del formulario hasta que el backend lo exponga.
3. **TipoEquipo dinámico:** Se obtiene de `GET /compras/tipos-equipo`. No hardcodear valores.
4. **Imagen solo en detalle:** La subida/preview de imagen se maneja dentro del modal de detalle, no en el formulario de crear/editar.
5. **Filtrado client-side:** Al no haber paginación, el listado completo se carga en memoria y se filtra/ordena en el frontend.
