# v0.9.0 — Gestión de Proveedores (CRUD) — Especificación de diseño

**VERSIÓN:** v0.9.0  
**SLUG:** proveedores-completo  
**PROMPT ORIGEN:** `Importante-main/AgenteDesigner/prompts-para-designer/v0.9.0-PROMPT-DESIGNER-proveedores-completo.md`  
**DEPENDENCIAS:** v0.8.0 (Compras) recomendada — si no hay API de proveedores, el listado se deriva de compras  
**BACKEND:** Ver `INFORME_BACKEND_PARA_FRONTEND.md` — §6.13 Proveedor (modelo anidado en Compra). **Hoy no hay CRUD dedicado**; esta spec define el diseño para cuando el backend lo exponga y cómo adaptar si no existe.

---

## 1. Alcance y usuarios

**Qué se diseña:** Vista de gestión de proveedores en `/proveedores` con listado, filtros, ordenamiento, modal crear/editar, modal detalle (con compras asociadas opcional) y eliminación con confirmación. La spec contempla **dos escenarios**: (A) backend con CRUD de proveedores; (B) sin API, listado derivado de proveedores únicos extraídos de las compras.

**Usuario(s) objetivo:**
- **ROLE_ADMIN:** CRUD completo incluyendo eliminar proveedores (si el backend lo restringe a ADMIN).
- **ROLE_USER:** Listar, crear, editar; sin eliminar si el backend lo restringe.

**Problema que resuelve:** Centralizar el mantenimiento de proveedores (nombre, CUIT, contacto, dirección, teléfono, email, observaciones) y poder ver las compras asociadas. Si no hay API, al menos mostrar los proveedores que ya existen vía compras y enlazar a Compras filtrado.

**Scope excluido (fuera de esta spec):**
- Paginación server-side (no asumida en el contrato).
- Subida de documentos por proveedor (no mencionado en el brief).

### 3.2 Modal Crear/Editar — Campos del formulario

| Campo UI | Campo API | Tipo | Obligatorio | Validación cliente |
|----------|-----------|------|-------------|-------------------|
| Nombre | nombre | text | Sí | No vacío (trim) |
| CUIT | cuit | text | No | — |
| Contacto | contacto | text | No | — |
| Dirección | direccion | text o textarea | No | — |
| Teléfono | telefono | text | No | — |
| Email | email | email | No | Si tiene valor, formato email válido |
| Observaciones | observaciones | textarea | No | — |

Layout: modal estándar (max-width 420px o 480px). Campos en una columna; opcionalmente dirección y observaciones en textarea para mejor UX.

### 3.3 Modal Detalle

**Sección 1 — Datos principales:**  
Filas etiqueta/valor para: Nombre, CUIT, Contacto, Teléfono, Email, Dirección, Observaciones. Valores vacíos mostrados como "—".

**Sección 2 — Compras asociadas (opcional):**
- Si hay API de proveedores y se puede obtener compras: listar compras donde `compra.proveedor.id === proveedor.id` (desde `GET /compras` en cliente, filtrado).
- Columnas resumidas: Fecha, Nº Factura, Importe, Tipo (o solo cantidad: "X compras").
- Botón/link **"Ver en Compras"** → navegar a `/compras` con query `?proveedorId=X` para que la vista de Compras (si soporta ese filtro) prefiltrar por proveedor.
- Si no hay compras para ese proveedor: texto "Sin compras asociadas".

**Acciones:** "Editar" (abre modal edición) + "Cerrar".

### 3.4 Eliminar proveedor

- Botón "Eliminar" en la fila (y opcionalmente en el detalle). Visible solo si el usuario tiene permiso (ej. ROLE_ADMIN si el backend restringe DELETE a ADMIN).
- Click → modal de confirmación: "¿Eliminar el proveedor [nombre]? Esta acción no se puede deshacer."
- Confirmar → `DELETE /proveedores/{id}`.
- Si el backend responde 400/409 con mensaje tipo "No se puede eliminar porque tiene compras asociadas", mostrar ese mensaje en el modal o en un toast de error.
- Éxito (204) → toast "Proveedor eliminado" → cerrar modal → recargar listado.

## 4. Estados y datos

### 4.1 Escenario A — Backend expone CRUD de proveedores

**Endpoints esperados:**

| Método | Ruta | Descripción | Roles sugeridos |
|--------|------|-------------|------------------|
| GET | `/proveedores` | Listar todos | ADMIN, USER |
| GET | `/proveedores/{id}` | Obtener uno | ADMIN, USER |
| POST | `/proveedores` | Crear (body CreateProveedorRequest) | ADMIN, USER |
| PUT | `/proveedores/{id}` | Actualizar (body UpdateProveedorRequest) | ADMIN, USER |
| DELETE | `/proveedores/{id}` | Eliminar (solo si no tiene compras o según regla de negocio) | ADMIN |

**Modelo Proveedor (INFORME_BACKEND §6.13):**

| Campo | Tipo | Notas |
|-------|------|--------|
| id | number | PK |
| nombre | string | Obligatorio |
| cuit | string \| null | |
| contacto | string \| null | |
| direccion | string \| null | |
| telefono | string \| null | |
| email | string \| null | |
| observaciones | string \| null | |

**Request crear/actualizar (mismo cuerpo):**  
nombre (obligatorio), cuit, contacto, direccion, telefono, email, observaciones (resto opcionales).

**Compras asociadas:**  
Obtener con `GET /compras` y filtrar en cliente por `compra.proveedor.id === id`, o usar un futuro `GET /compras?proveedorId=X` si el backend lo añade.

### 4.2 Escenario B — Sin API de proveedores

**Fuente de datos:**
- Listado: `GET /compras` → extraer proveedores únicos por `proveedor.id` (mantener orden por nombre A–Z o por primera aparición).
- Detalle: mismo objeto proveedor; compras asociadas = compras donde `compra.proveedor.id === proveedor.id`.
- Crear / Editar / Eliminar: **no disponibles**. El botón "Nuevo proveedor" puede estar oculto, deshabilitado con tooltip "Próximamente cuando el backend exponga el API de proveedores", o mostrar un mensaje informativo al hacer click. Idem "Editar" y "Eliminar": ocultos o deshabilitados con mensaje claro.

**Cálculo de "Compras" por fila:**  
Para cada proveedor, contar compras donde `compra.proveedor.id === p.id`. Mostrar el número en la columna "Compras" o "—" si no se puede calcular.

### 4.3 Estados de UI

| Estado | Cuándo | Comportamiento |
|--------|--------|----------------|
| Loading | GET /proveedores (o GET /compras en B) en curso | Spinner + "Cargando proveedores..." |
| Vacío | 0 proveedores | A: "No hay proveedores registrados" + CTA "Crear primer proveedor". B: "No hay proveedores registrados. Los proveedores aparecerán al cargar compras." (sin CTA crear o con enlace a Compras) |
| Sin resultados | Filtros activos y 0 coincidencias | "No se encontraron proveedores con los filtros aplicados" + "Limpiar filtros" |
| Error | Fallo de petición | Mensaje de error + "Reintentar" |
| Con datos | Lista cargada | Tabla + filtros operativos |
| Modal guardando | POST/PUT en curso | Spinner en botón, campos deshabilitados |
| Error API (400/409/500) | Respuesta de backend con error | Banner en modal o toast de error; mensaje del cuerpo cuando exista |

## 5. Flujos de usuario

### 5.1 Flujo principal — Listar y filtrar (Escenario A)

1. Usuario entra a `/proveedores`.
2. `GET /proveedores` al montar → loading → listado.
3. Filtro por texto libre: busca en nombre, CUIT, contacto, email, observaciones (client-side).
4. "Limpiar filtros" resetea búsqueda y filtros opcionales.
5. "Mostrando X de Y proveedores" se actualiza según filtros.
6. Ordenamiento: click en header de columna (Nombre, CUIT, Contacto) → alternar ASC/DESC. Default: nombre ASC (A–Z).

### 5.2 Flujo — Crear proveedor (solo Escenario A)

1. Click "+ Nuevo proveedor" → abre modal.
2. Completar al menos Nombre (obligatorio). Opcional: CUIT, contacto, dirección, teléfono, email, observaciones.
3. Si email tiene valor, validar formato.
4. "Guardar" → validar → POST /proveedores → toast "Proveedor creado" → cerrar modal → recargar listado.
5. Error 400/409/500 → mostrar mensaje en banner del modal.

### 5.3 Flujo — Editar proveedor (solo Escenario A)

1. Click "Editar" en fila o desde detalle → abre modal con datos precargados.
2. Modificar campos → "Guardar" → PUT /proveedores/{id} → toast "Proveedor actualizado" → cerrar → recargar.

### 5.4 Flujo — Ver detalle

1. Click "Ver" (o en el nombre si es link) → abre modal detalle.
2. Se muestran todos los campos del proveedor.
3. Opcional: lista de compras asociadas (filtradas en cliente desde GET /compras) y botón "Ver en Compras" → `/compras?proveedorId=X`.
4. "Editar" → cierra detalle y abre modal edición.

### 5.5 Flujo — Eliminar (solo Escenario A, solo ADMIN si aplica)

1. Click "Eliminar" → modal confirmación con nombre del proveedor.
2. Confirmar → DELETE /proveedores/{id} → éxito → toast "Proveedor eliminado" → recargar.
3. Si backend responde error (ej. "No se puede eliminar porque tiene compras asociadas") → mostrar mensaje en modal o toast.

### 5.6 Flujo — Escenario B (sin API)

1. Al montar: `GET /compras` → extraer proveedores únicos → mostrar en la misma tabla.
2. Filtros y ordenamiento funcionan igual sobre esa lista.
3. "Nuevo proveedor" / "Editar" / "Eliminar" ocultos o deshabilitados con mensaje claro.
4. "Ver" y "Ver en Compras" siguen disponibles usando los datos derivados.

### Validación — Formulario crear/editar

| Campo | Condición de error | Mensaje |
|-------|-------------------|---------|
| nombre | Vacío (tras trim) | "El nombre es obligatorio." |
| email | Con valor pero formato inválido | "Ingresá un email válido." |

- [ ] La vista `/proveedores` muestra un listado de proveedores (desde API o derivado de compras en B).
- [ ] Filtro por texto libre busca en nombre, CUIT, contacto, email, observaciones.
- [ ] "Limpiar filtros" resetea todos los filtros.
- [ ] "Mostrando X de Y proveedores" es correcto.
- [ ] Se puede ordenar por columna (al menos Nombre; recomendado CUIT, Contacto).
- [ ] Orden por defecto es por nombre A–Z.

### CRUD (Escenario A)
- [ ] Se puede crear un proveedor con todos los campos (POST).
- [ ] Se puede editar un proveedor (PUT).
- [ ] Solo ADMIN puede eliminar si el backend lo restringe; el botón no se muestra a USER.
- [ ] Modal de confirmación antes de eliminar.
- [ ] Toast en éxito al crear, editar y eliminar.
- [ ] Si el backend rechaza DELETE por compras asociadas, se muestra el mensaje al usuario.

### Detalle
- [ ] Se puede abrir el detalle de un proveedor (modal).
- [ ] Se muestran todos los campos del proveedor (nombre, CUIT, contacto, teléfono, email, dirección, observaciones).
- [ ] Opcional: se listan compras asociadas y/o enlace "Ver en Compras" a `/compras?proveedorId=X`.

### Estados y permisos
- [ ] Loading con spinner.
- [ ] Estado vacío con mensaje y CTA adecuado al escenario (A o B).
- [ ] Estado error con Reintentar.
- [ ] Sin resultados con "Limpiar filtros".
- [ ] Permisos según rol (Eliminar solo ADMIN si aplica).

### Escenario B (sin API)

- [ ] Crear / Editar / Eliminar deshabilitados o con mensaje claro ("Próximamente" o "Requiere API de proveedores").
- [ ] Detalle y "Ver compras" (o columna Compras) funcionan con los datos disponibles.

### Calidad
- [ ] Build sin errores.
- [ ] Sin errores en consola.
- [ ] Responsive (filtros apilados y tabla con scroll horizontal en &lt; 768px).
- [ ] Textos en español.

**Supuestos y abiertos:**

1. **Existencia del API:** Si al implementar el backend aún no expone `/proveedores`, el programador implementa Escenario B (listado desde compras, CRUD deshabilitado) siguiendo la misma UI.
2. **DELETE con compras:** Si el backend no permite eliminar un proveedor con compras, se muestra el mensaje de error devuelto (cuerpo de la respuesta).
3. **Filtro "Con compras" / "Sin compras":** Opcional; solo si hay datos para derivarlo (en B se puede: proveedor tiene compras si aparece en el array de compras). Si no se implementa, no es bloqueante.

5. **CUIT:** Mostrar como texto plano tal como viene del backend; si en el futuro el backend envía formato 30-71234567-9, mostrarlo así.
